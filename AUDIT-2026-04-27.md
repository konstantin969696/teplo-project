# Аудит teplo-project — 27.04.2026

## КРИТИЧЕСКОЕ (баги, data loss риски)

### F01 — Несогласованные температуры в EquipmentResultsTable (legacy global temps)
**Где:** `src/components/equipment/EquipmentResultsTable.tsx:121-122`
**Что:** Компонент читает `tSupply`/`tReturn` из `useProjectStore` (legacy v1.0 поля), а не из `HeatingSystem`. После миграции v1.1 эти поля всегда `undefined` → fallback 80/60. В проекте с несколькими системами (например, низкотемпературная 55/45) таблица итогов покажет неверный LMTD и неверные Q_факт / запас%.
**Почему важно:** Пользователь принимает инженерные решения по таблице итогов; неверные цифры → недобор/перебор приборов.
**Как чинить:** Заменить на `useSystemStore` + `equipment.systemId`, аналогично `EquipmentRow`/`EquipmentSubRow`.

### F02 — `as unknown as` в `handleAddFirstEquipment` и `handleAddEquipment` скрывает несоответствие типов
**Где:** `src/components/equipment/EquipmentRow.tsx:145-159`, `src/components/equipment/EquipmentSubTable.tsx:41-56`
**Что:** При создании `Equipment` через `addEquipment` используется `as unknown as Omit<Equipment, 'id'>`, хотя `systemId` обязателен с v1.1. Если в будущем добавится ещё одно обязательное поле, компилятор не поймает ошибку.
**Почему важно:** Любое расширение `Equipment` станет land-mine: runtime-объект не будет соответствовать типу, а TS промолчит.
**Как чинить:** Убрать `as unknown as`, собрать объект строго по `Omit<Equipment, 'id'>`, или ввести фабричную функцию `createDefaultEquipment(roomId, systemId)`.

### F03 — `clearLegacyV10Fields` использует `as unknown as` и `replace=true` без гарантии типа
**Где:** `src/store/projectStore.ts:44-54`
**Что:** `set(state => { ...rest } as unknown as ProjectState, true)` — `replace=true` заменяет состояние, но `rest` теряет методы-экшены (addRoom и т.д.), потому что в `partialize` они не сохраняются, а в `set` с `replace` Zustand не мержит экшены автоматически. В теории после вызова store может остаться без методов.
**Почему важно:** Data-loss риск: если `replace=true` сработает не так, как ожидается, пользователь потеряет возможность редактировать проект до перезагрузки.
**Как чинить:** Вместо `replace=true` делать shallow-merge с явным удалением ключей: `return { ...state, tSupply: undefined, ... }` (Zustand игнорирует `undefined` при merge, но для надёжности лучше использовать `delete` паттерн или вынести очистку в отдельный экшен после миграции).

### F04 — `useGckPath` вычисляет `equipmentQMap` в `useMemo` с огромным массивом зависимостей
**Где:** `src/components/hydraulics/useGckPath.ts:66-167`
**Что:** `useMemo` зависит от `equipment`, `equipmentOrder`, `rooms`, `tOutside`, `enclosures`, `enclosureOrder`, `systemsMap`, `catalogModels` — почти всё приложение. Любое изменение комнаты/ограждения/прибора пересчитывает гидравлику всех систем. Кроме того, `kmsCatalog` передаётся как `Object.values(s.elements)` — новый массив каждый рендер, но он не в deps, что нарушает правила React (хотя lint может не ловить).
**Почему важно:** Лаг UI при >50 сегментах; на мобильных устройствах фризы.
**Как чинить:** Вынести `equipmentQMap` в отдельный селектор/memo с `useShallow` по `equipmentOrder` + `equipment`, и пересчитывать только при изменении приборов. Для `kmsCatalog` использовать `useShallow` и добавить в deps.

### F05 — `useSystemSummary` читает `segments` целиком, а не фильтрует по systemId
**Где:** `src/hooks/useSystemSummary.ts:19-30`
**Что:** `const segments = useSegmentStore(s => s.segments)` подписывает компонент на **все** изменения любого сегмента. `count` фильтрует внутри `useMemo`, но сам факт изменения сегмента в другой системе вызовет re-render.
**Почему важно:** В проекте с 5+ системами accordion каждой системы будет перерисовываться при редактировании любой другой.
**Как чинить:** Использовать `useShallow` + селектор, возвращающий только `segmentOrder` и количество по `systemId`.

## ВАЖНОЕ (техдолг, архитектура)

### F10 — Дублирование логики вычисления Q_пом: EquipmentRow, EquipmentResultsTable, useGckPath, SummaryTab
**Где:** `src/components/equipment/EquipmentRow.tsx:102-114`, `src/components/equipment/EquipmentResultsTable.tsx:134-147`, `src/components/hydraulics/useGckPath.ts:91-119`, `src/components/summary/SummaryTab.tsx:63-84`
**Что:** Четыре места независимо вызывают `getEngineWorker().heatLossForRooms(...)` (или `calculateRoomTotals` напрямую) для одних и тех же данных. Результаты не кэшируются между компонентами.
**Почему важно:** Расход CPU, race conditions (разные компоненты могут получить разные результаты если данные изменятся между вызовами), сложность отладки.
**Как чинить:** Ввести единый derived store (или React Context + memo) для `roomId → RoomHeatLossResult`, который обновляется один раз на изменение rooms/enclosures/city.

### F11 — `useSegmentTree` и `useGckPath` дублируют построение childrenMap
**Где:** `src/hooks/useSegmentTree.ts:15-22`, `src/components/hydraulics/useGckPath.ts:126-130`, `src/components/hydraulics/SegmentTable.tsx:37-49`
**Что:** `buildSegmentTree` строит `childrenMap` внутри себя; `useGckPath` строит свой `childrenMap` для `derivedQ`; `SegmentTable` строит `parentChain` для collapse-логики. Три разных O(n) прохода по одним сегментам.
**Почему важно:** Не критично при <200 сегментах, но архитектурный запах: дерево — центральная структура, а строится ad-hoc.
**Как чинить:** Вынести `useSegmentTreeData(systemId)` hook, возвращающий `{ tree, childrenMap, parentChain }` в одном `useMemo`.

### F12 — `deriveEquipmentQActual` содержит бизнес-логику подбора приборов в UI-хелпере
**Где:** `src/components/equipment/equipment-help.ts:73-132`
**Что:** Функция выбирает между panel/convector/sectional, ищет варианты в каталоге, считает секции — это чистая бизнес-логика. Она живёт в `components/equipment/`, а не в `engine/`.
**Почему важно:** Нельзя переиспользовать в Web Worker, PDF-экспорте или тестах engine без импорта UI-слоя.
**Как чинить:** Перенести `deriveEquipmentQActual` и `getEffectiveNExponent` в `engine/equipmentSelection.ts` (или `engine/equipment.ts`), оставив в `equipment-help.ts` только UI-форматтеры.

### F13 — `applyImportedStores` не валидирует вложенные поля сегментов/приборов
**Где:** `src/services/importService.ts:18-88`
**Что:** `shapeMerge` проверяет только верхний уровень (`segments: 'record'`), но не проверяет, что каждый `Segment` содержит обязательные поля `systemId`, `kind`, `autoGenerated`. Импорт повреждённого JSON может создать сегменты без `systemId` → краш при открытии гидравлики.
**Почему важно:** Пользователь может импортировать руками отредактированный JSON; missing `systemId` вызовет `undefined` в `filterSegmentsBySystem` и далее в `derivedQ`.
**Как чинить:** Добавить sanitize-проход: для каждого сегмента `systemId ||= ''`, `kind ||= 'branch'`, `autoGenerated ||= false`; для equipment `systemId ||= orderedSystems[0]?.id || ''`.

### F14 — `buildSummaryDocument` вызывает `buildHeatLossDocument`/`buildEquipmentDocument` и т.д., каждый из которых заново читает все сторы
**Где:** `src/export/content/builders.ts:230-251`
**Что:** `buildSummaryDocument` вызывает 4 builder-функции, каждая делает `useXxxStore.getState()` — 4 полных снимка состояния. `buildHeatLossDocument` заново считает `calculateRoomTotals` для всех комнат, хотя `SummaryTab` уже сделал это в `useEffect`.
**Почему важно:** PDF-экспорт большого проекта будет тормозить; дублирование вычислений.
**Как чинить:** Передавать предварительно вычисленные `RoomHeatLossResult[]` и `equipmentList` в builders как параметры, или ввести `ExportContext` с мемоизированными derived данными.

### F15 — `EquipmentResultsTable` не учитывает per-system температуры (D-27)
**Где:** `src/components/equipment/EquipmentResultsTable.tsx:121-122`, `computeRow:67`
**Что:** `computeRow` принимает единые `tSupply`/`tReturn` для всей таблицы. Если в комнате есть приборы из разных систем (допустимо по D-10), `EquipmentResultsTable` посчитает их все с температурой первой системы.
**Почему важно:** Нарушает фичу D-27 (per-equipment system temps); итоговая таблица врёт.
**Как чинить:** В `computeRow` для каждого прибора брать `tSupply`/`tReturn` из его `systemId`, как это делает `EquipmentRow`.

### F16 — `useUfhSystemTemps` и `useEquipmentSystemTemps` используют `as unknown` cast
**Где:** `src/hooks/useUfhSystemTemps.ts:17`, `src/hooks/useEquipmentSystemTemps.ts:17`
**Что:** `(s.loops[loopId] as unknown as { systemId?: string | null })?.systemId` — вместо того чтобы типизировать `UfhLoop` с обязательным `systemId` (он обязателен с v1.1), код обходит типизацию.
**Почему важно:** Если `systemId` станет truly required, эти cast'ы останутся и скроют реальные баги.
**Как чинить:** Обновить тип `UfhLoop` — `systemId: string` (не `string | null`?) и убрать cast; добавить миграцию backfill для legacy данных.

### F17 — `safeStorage.setItem` молча проглатывает ошибки, кроме `QuotaExceededError`
**Где:** `src/store/safeStorage.ts:18-25`
**Что:** `try/catch` ловит любое исключение, но toast показывает только при `QuotaExceededError`. Если `localStorage.setItem` бросит `SecurityError` (private mode Safari) или `TypeError`, пользователь не узнает, что данные не сохранились.
**Почему важно:** Data loss: пользователь думает, что проект сохранён, а на самом деле localStorage отключён.
**Как чинить:** Добавить `else` ветку с `toast.error` для всех остальных ошибок + fallback на `console.error`.

## МИНОРНОЕ (стиль, naming, незначительное)

### F30 — Дублирование `SCHEMA_LABEL`/`PIPE_MATERIAL_LABELS` между `hydraulics-help.ts` и `SummaryTab.tsx`
**Где:** `src/components/hydraulics/hydraulics-help.ts:9-23`, `src/components/summary/SummaryTab.tsx:20-26`
**Что:** Два одинаковых (почти) словаря меток. `SummaryTab` добавляет устаревшие ключи (`one-pipe-vertical` и т.д.), которых нет в `SchemaType`.
**Почему важно:** Рассинхронизация меток при добавлении новой схемы.
**Как чинить:** Импортировать `SCHEMA_LABELS` из `hydraulics-help.ts` в `SummaryTab`.

### F31 — `SegmentTable` строит `parentChain` через `tree.find(...)` в цикле — O(n²)
**Где:** `src/components/hydraulics/SegmentTable.tsx:37-49`
**Что:** Для каждого узла `tree.find(n => n.segment.id === parentId)` сканирует весь массив. При 200 сегментах это 40 000 операций.
**Почему важно:** Микро-фриз при открытии таблицы гидравлики на большом проекте.
**Как чинить:** Построить `Map<string, SegmentTreeNode>` за O(n) один раз и использовать его для lookup.

### F32 — `KIND_LABELS`/`CONNECTION_LABELS`/`INSTALLATION_LABELS` используют `Record<string, string>` вместо строгих union
**Где:** `src/components/equipment/equipment-help.ts:36-54`
**Что:** `Record<string, string>` позволяет передать любой ключ без TS-ошибки. Должно быть `Record<EquipmentKind, string>` и т.д.
**Почему важно:** Типобезопасность: при добавлении нового `EquipmentKind` компилятор не подскажет, что label не добавлен.
**Как чинить:** Заменить типы на strict unions.

### F33 — `inputClass` продублирован в `RoomRow.tsx`, `EnclosureRow.tsx`, `equipment-help.ts`, `hydraulics-help.ts`, `ufh-help.ts`
**Где:** `src/components/heatLoss/RoomRow.tsx:26`, `src/components/heatLoss/EnclosureRow.tsx:37`, `src/components/equipment/equipment-help.ts:13`, `src/components/hydraulics/hydraulics-help.ts:7`, `src/components/ufh/ufh-help.ts:7`
**Что:** Один и тот же CSS-класс скопирован в 5 файлов.
**Почему важно:** Изменение стиля input требует правки в 5 местах.
**Как чинить:** Вынести в `src/styles/inputClass.ts` или CSS-utility.

### F34 — `useGckPath` не обрабатывает `catch` от worker promise
**Где:** `src/components/hydraulics/useGckPath.ts` — нет `.catch()` на `getEngineWorker()` вызовы, но тут worker используется синхронно (внутри `useMemo`).
**Что:** На самом деле `useGckPath` не использует worker вообще — всё считает в главном потоке. Комментарий в `useGckPath.ts` говорит "Phase 6 Web Worker", но код синхронный.
**Почему важно:** Не критично, но комментарий/название вводит в заблуждение.
**Как чинить:** Убрать устаревший комментарий или реализовать фактический offloading в worker.

### F35 — `UfhLoopRow` создаёт loop через `ensureLoop()` при каждом взаимодействии, даже если loop уже есть
**Где:** `src/components/ufh/UfhLoopRow.tsx:83-106`
**Что:** `ensureLoop()` вызывается в `handleAreaCommit`, `onChange` покрытия, трубы, шага — каждый раз делает `if (loop) return loop`, но сам факт наличия функции с side-effect (addLoop) в event handler'ах запутывает.
**Почему важно:** Не баг, но архитектурный запах: UI-компонент решает, создавать ли entity в store.
**Как чинить:** Вынести логику "создать loop если нет" в `useUfhLoopStore` action (например, `ensureLoopForRoom(roomId)`), чтобы UI только вызывал `updateLoop`.

### F36 — `buildSummaryDocument` и `SummaryTab` дублируют `SCHEMA_LABEL`
**Где:** `src/export/content/builders.ts:19-25`, `src/components/summary/SummaryTab.tsx:20-26`
**Что:** Два одинаковых словаря.
**Почему важно:** Рассинхронизация при изменении labels.
**Как чинить:** Импортировать из единого источника (`hydraulics-help.ts` или `types/hydraulics.ts`).

### F37 — `calculatePoolEvaporationHeat` не защищён от `NaN` в `pool.tWaterC` / `pool.phi`
**Где:** `src/engine/poolEvaporation.ts:68-78`
**Что:** Нет проверки `Number.isFinite(pool.tWaterC)` или `phi`. Если пользователь введёт `NaN` (например, очистив поле), `calculateSaturatedVaporPressure` вернёт `NaN`, и весь `Q_исп` станет `NaN`, что просочится в `qTotal`.
**Почему важно:** `NaN` заражает все downstream вычисления; в UI отобразится пустая ячейка без объяснения.
**Как чинить:** Добавить guard: `if (!Number.isFinite(pool.tWaterC) || !Number.isFinite(pool.phi)) return 0`.

### F38 — `getEngineWorker()` создаёт worker один раз, но никогда не терминирует
**Где:** `src/workers/useEngineWorker.ts:6-12`
**Что:** Worker создаётся при первом вызове и живёт до закрытия вкладки. Если приложение переключает проекты часто, worker держит ссылку на старые данные через замыкания (не критично, т.к. state передаётся явно, но память растёт).
**Почему важно:** В долгоживущей вкладке с множеством импортов/экспортов — утечка памяти.
**Как чинить:** Добавить `terminate()` при размонтировании App или использовать `WeakRef` / пул workers.
