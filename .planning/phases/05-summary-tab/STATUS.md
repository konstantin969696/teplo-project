# Phase 05 — Tab «Сводка» — STATUS

**Status:** 🟢 closed (2026-04-26)

## Что было

В `App.tsx` для tab 4 (Сводка) рендерился `EmptyTabState` — заглушка. Никаких сводных данных по объекту не показывалось.

## Что сделано

`src/components/summary/SummaryTab.tsx` — single-file компонент, read-only представление 5 секций:

1. **Объект** — город, t наружная, t внутри, ΔT, ГСОП, зона влажности, кол-во помещений и систем.
2. **Теплопотери** — таблица помещений (этаж, площадь, Q_осн, Q_инф, Q_вент, Q_итого) + итог по объекту в Вт и кВт. Использует `calculateRoomTotals` из engine.
3. **Системы отопления** — карточки по каждой системе (схема, t подачи/обратки, тёплый пол t, теплоноситель, материал труб). Лейблы schemaType — на русский.
4. **Приборы отопления** — таблица (помещение, тип, подключение, установка, модель). Кол-во в заголовке.
5. **Тёплый пол** — таблица петель (помещение, F_тп, шаг, покрытие, труба). Только enabled.

Внизу — нормативная подпись (СП 50.13330.2012, СП 60.13330.2020, СП 131.13330.2020).

## Архитектурные заметки

- **Не создавал собственного state.** Все данные тянутся из существующих stores (`useProjectStore`, `useSystemStore`, `useEnclosureStore`, `useEquipmentStore`, `useUfhLoopStore`).
- **Не использовал готовые селекторы вроде `selectOrderedSystems`** — они возвращают новый array на каждом render и подписка через `useStore(selector)` лупит "Maximum update depth exceeded". Подписка на сырые поля + `useMemo` для производных — стабильная схема.
- **Engine — только `calculateRoomTotals`.** Hydraulics-tree не строится в сводке (это работа отдельной фазы, потенциально перед PDF — там нужен полный расчёт ΔP по магистрали, цикло-кольцо и т.д.). Сейчас гидравлические числа в сводке не показаны, но в фазе 06/07 (экспорт) станут необходимы.

## Что НЕ покрыто (для будущих фаз)

- **Гидравлика** в сводке — расход теплоносителя, главное циркуляционное кольцо, ΔP. Когда будет нужно (фаза 06 «Экспорт PDF» или раньше), доделать.
- **UFH расчёты** — длина петли, q_удельный, t_пола, ΔP петли. Сейчас только конфигурация.
- **Сборка по системам** — приборы и петли сейчас в плоских таблицах. Можно сгруппировать по `systemId` если систем несколько (сейчас система одна по миграции — выглядит избыточно).
- **Группировка помещений по этажам** (как в RoomsTable / ResultsTable) — пока плоский список.
